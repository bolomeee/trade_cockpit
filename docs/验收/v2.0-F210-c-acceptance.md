## 数据一致性抽查（可选）

打开 Chrome DevTools → Network → 找 `/api/ai/trade_plan`，对照：

```
Request body.input:
  ticker, setupType, setupQuality, entry, stop, target2r, target3r,
  size, rewardRisk, accountRiskPct, earningsRisk, deterministicHash
```

与 backend `app/ai/schemas/trade_plan.py` 的 `TradePlanInput` 字段名严格一致（**注意三处重命名**：entryPrice→entry / stopPrice→stop / suggestedShares→size）。

不应包含：`effectiveRiskPct`, `regimeCap`, `userSettingCap`, `earningsDate`, `riskPerShare`, `positionValue`（schema `extra: forbid`，多了会 422）。

Response body 关键校验：
- `output.entry === input.entry`
- `output.stop === input.stop`
- `output.size === input.size`
（任一不等 backend 应直接 409，不会到达前端 200 路径）

### 阻断 Bug：earningsRisk=null 时 trade_plan 100% 422

**现象**：
- 选任意当前 ready 候选股（VALE / AAPL / CIM）→ 点 `Generate AI Plan` → 永远显示"AI 暂不可用"
- 网络请求 `POST /api/ai/trade_plan` 返回 **422 VALIDATION_ERROR**

**根因**：
1. backend [trade_plan.py:48](backend/app/ai/schemas/trade_plan.py:48) 定义 `earningsRisk: Literal["SAFE","CAUTION","DANGER"]` — **不允许 null**
2. `GET /api/cockpit/decision/{ticker}` 在没有 earnings 数据时返回 `earningsRisk: null`（已验证 VALE/AAPL/CIM 全为 null）
3. 前端 [AiTradePlanSection.tsx:48](frontend/src/cockpit/components/AiTradePlanSection.tsx:48) `buildTradePlanInput` 直传 `decision.earningsRisk`，未做 null 兜底
4. data-mapping 与 schema 之间存在 null 语义错位

**为什么单元/集成测试没抓到**：
- `DecisionPanelWidget.test.tsx` §T 的 mockDecision 用的是 `earningsRisk: 'SAFE'` 等枚举值
- 没有 mock decision data 中 `earningsRisk: null` 的 case
- Sprint Contract §7 风险表只列了 `setupQuality null` 的 typing 问题，遗漏了 `earningsRisk null` schema 校验问题

**影响范围**：当前生产环境下 100% 用户无法使用 AI Trade Plan（所有 ticker 的 earningsRisk 都是 null）。

**修复路径建议**（需用户决策一选其一）：

| 方案 | 改动点 | 影响 |
|------|-------|------|
| A. backend 接受 null | [trade_plan.py:48](backend/app/ai/schemas/trade_plan.py:48) 改 `Literal[...] \| None`；guardrail / system prompt 处理 null | 最小改动；语义上"无 earnings 风险数据" = "未知"，AI 默认按 SAFE 处理 |
| B. 前端兜底为 SAFE | [AiTradePlanSection.tsx:48](frontend/src/cockpit/components/AiTradePlanSection.tsx:48) `decision.earningsRisk ?? 'SAFE'` | 前端单文件改，但语义上"无数据"≠"安全"，可能误导 AI memo |
| C. decision API 不返回 null | backend 有 earnings 数据时 SAFE/CAUTION/DANGER，无数据时也返回 SAFE | 改动 cockpit decision service，影响面更大 |

⚠️ **建议方案 A** — 与 schema `setupQuality: Literal[...] \| None` 同样语义对称，且 backend 才是字段权威（`earningsRisk` 在 DATA-MODEL.md 中允许 null）。
