## 验收发现的问题与处理

### 🔴 P0 修复：trendScore 阈值不匹配实际值域

**发现**：开发用 `trendScore >= 60 → up / <= 40 → down` 阈值，但后端 `trend_score` 是 0-5 整数（MA 排列阶梯：close>MA10>MA21>MA50>MA150>MA200，定义在 `backend/app/services/cockpit/setup_service.py:_compute_trend_score`）。线上 35 行实测值域 ∈ {0,1,2,3,4}，原阈值会把所有行映射为 'down'，导致 LLM 收到错误前提（如 VALE BREAKOUT 被解读为 "in a downtrend... false break risk"）。

**根因**：Sprint Contract Q1 已识别此风险（"trendScore 实际值域不是 0-100" 概率中、影响"映射全失准"），默认采取在 acceptance 阶段实测验证。本次验收触发了该路径。

**修复**（commit 192d7f5）：
- `AiSetupExplainerPopover.tsx`：阈值改为 `>=4 → up / <=1 → down / else sideways`
- 测试 fixtures 同步到 0-5 值域
- 新增 §S8b 6 个边界测试覆盖（trendScore=0..5 → trend mapping）
- 95/95 tests pass

**验证**：浏览器实测 AAPL（trendScore=3）→ trend='sideways'，AI 输出 "AAPL — Sideways trend + Breakout setup" + Quality B，与实际 setup 状态一致。

实测窄视口下表格被压窄是 RGL 容器宽度问题，不是 `?` 列引起。正常宽度下无碍。
